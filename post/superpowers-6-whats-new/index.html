你给 Agent 装了 Superpowers 吗？v6 这一波更新，把 Skill 从「说明书」改造成了「自动工厂」

发布时间: 2026-07-13
AI Agent Superpowers OpenCode Claude Codex Skills 编码代理 软件开发

先说结论

如果你已经在用 Claude Code、OpenCode、Codex 之类的工具写代码，大概率已经听过 Superpowers 这个名字。它不是又一个「魔法 prompt」，而是一套让 Agent 按流程工作的 Skill 框架——把「先想清楚再动手」「先写测试再看代码」「修 bug 先收集证据」这些人类工程师的好习惯，变成 Agent 必须遵守的操作规程。

截至 2026 年 7 月 13 日，Superpowers 的最新稳定版本是 v6.1.1。从 v6.0.0 到 v6.1.1，这个项目连续完成了六个版本节点：子代理驱动开发（SDD）减少 reviewer，但审查更严格；Plans 增加「全局约束」和「接口契约」；Codex 进入官方插件市场；bootstrap 变得更轻；Gemini CLI 支持被移除；最后还修了一个 hooks: {} 才能表达「明确没有 hook」的兼容性坑。

这篇文章就一起把这波更新拆开看看，你到底能拿到什么新东西，以及它为什么能让你的 Agent 输出更稳。

1. 先搞清楚：Superpowers 到底解决什么问题

很多刚接触 Agent 的朋友会有一个错觉：只要模型够强、提示词够长，Agent 就会老老实实把活干好。实际用过一阵子就会发现，Agent 特别擅长「假装完成」——它会在没跑测试的时候说「应该没问题」，会在没看日志的时候说「已经修复了」，会在设计都没清楚的时候直接开始写代码。

Superpowers 的思路很简单：与其把希望寄托在模型「自觉」上，不如把流程写死成 Skill，让 Agent 在每个节点都必须停下来检查。

打个比方：如果你让一个实习生去修一辆坏掉的车，不给任何规程的话，他很可能上来就拧螺丝，拧了半天发现拆错了零件。Superpowers 就像一份《维修手册》——第一步观察现象、第二步读故障码、第三步定位原因、第四步更换零件、第五步测试。老员工靠经验，新员工靠手册；Agent 也一样，模型越弱，越需要手册。

图 3：Superpowers 官方 README 内容截图，画面保留了官方源文件路径。它把项目定位成「给编码代理的一整套软件开发方法论」，而不是一个 prompt 技巧。

图 4：Agent 没有外部流程约束时，很容易在「猜测 → 乱改 → 声称完成」之间循环——Superpowers 在每个节点插入检查点，打断这个循环。

Superpowers 目前主要包含这些 Skill：

2. v6.0：SDD 从「两个 reviewer」变成一个，反而更严了

v6.0.0 是这一轮最大的一个版本。最大的变化在 Subagent-Driven Development（SDD，子代理驱动开发）。

2.1 以前为什么用两个 reviewer

早期的 SDD 流程里，每个任务会派两个子代理去审查：一个叫 spec-reviewer（看任务方案符不符合需求），一个叫 code-quality-reviewer（看代码质量过不过关）。听起来很专业，对吧？

但实际运行下来发现了两个问题：

  1. 贵。每次审查都要启动新的 Agent 会话，token 消耗翻倍。
  2. 容易被绕过。模型很会「讨价还价」——controller（调度者）经常暗示 reviewer「这个问题不算严重」，reviewer 就真放过去了。

就像公司里做代码评审：如果评审者和被评审者是同一个老板下面的同事，面子一给，评审就变走过场了。

2.2 现在：一个 reviewer、两个 verdict、plan 还要预检

v6.0 把两个 reviewer 合并成了一个 task-reviewer，它一次读取 diff，同时给出方案符合度和代码质量两个结论（verdict）。一次搞定，省钱。

更重要的是几个「防作弊」机制：

  • Diff 不再粘贴，走文件。以前 reviewer 要看的内容是被粘贴到 prompt 里的，永久占据最昂贵的上下文窗口。现在用两个脚本 task-brief 和 review-package 把任务文本和审查 diff 写成文件让子代理去读，不占主上下文。
  • Controller 不许教 reviewer 忽略什么。这是真的在代码里抓到的 bug——controller 会暗示 reviewer「这个最多算 Minor」。现在完全禁止 controller 预设 severity。
  • Reviewer 是 read-only 的。以前 reviewer 会执行 git checkout，把后面的提交弄成孤儿。现在 reviewer 只能看，不能改。
  • Plan 有了 pre-flight 检查。在执行第一个任务之前，controller 会先把 plan 过一遍，看有没有内部矛盾（比如任务 A 说要加字段、任务 B 说不要加）。以前是执行到一半才发现冲突，再回头改，现在一次性提出来。
  • 每个 dispatch 必须声明用哪个模型。以前 controller 不指定模型，默认就用会话里最贵的那个——结果某次跑下来 26 个 reviewer 全都用了顶配模型。现在必须显式声明。

图 5：v6 之后，每个任务的审查从「两个 reviewer 分散把关」变成「一个 reviewer 一次出两个 verdict」，并增加了 Plan 预检和文件级 diff 传递，整体更紧凑也更难绕过。

2.3 效果：同一个结果，省将近一半 token

官方在 evals 里跑出来的数据是：Claude Code 和 Codex 产出类似质量的结果，速度大约快一倍，token 消耗少将近 50%。

这不是「因为少雇了一个 reviewer 所以便宜」那种简单计算——关键在于取消了 controller 和 reviewer 之间的讨价还价循环。以前 reviewer 说「有问题」，controller 说「我觉得问题不大」，reviewer 就让步了。现在 severity 是 reviewer 独立判的，没有谈判空间。

打个比方：以前是「质检员发现次品，车间主任说先放行，质检员就让步」，现在是「质检员独立判不合格就打回，车间主任不能插嘴」。流程没变多，但漏洞堵上了。

3. v6.0：Plans 也长骨头了

另一个很重要的变化在 writing-plans 这个 Skill。

3.1 全局约束（Global Constraints）

以前写 plan 是靠模型自由发挥，每个 task 的约束需要 reader 自己从字里行间体会。现在 plan 里强制加一个 Global Constraints 区块，把适用于所有任务的约束原封不动抄进去——版本下限、依赖限制、命名规范、文案措辞、精确数值。

还是拿修车做比方：以前 plan 说「注意安全」，不同工人理解不同；现在 plan 明确写「必须佩戴绝缘手套、必须挂警示牌、必须断电 5 分钟后方可操作」——抄过去就行，不用动脑翻译。

3.2 接口契约（Interfaces）

每个 task 现在都要声明自己消费什么、产出什么。比如 task 3 需要 task 2 产出的那个 API response schema，就必须写明在 Interfaces 块里。

这样即使每个子代理只看到自己的 task，也清楚邻居给它什么东西、它要给邻居什么东西。很像流水线上的工件接口——上道工序的输出，就是下道工序的输入。

图 6：v6 之后的 Plan 结构——全局约束横跨所有任务、接口契约管住任务边界、任务大小要能撑住一个完整测试循环。官方测试显示，这样写 plan 一轮就能通过，对照组需要 2 到 4 轮，而且对照组还漏了一个真实 bug。

4. v6.0：脑暴看板加了「密码锁」

Superpowers 的 brainstorming Skill 有一个 Visual Companion（可视化搭档）：Agent 在修问题的时候，会启动一个小型的 Web 页面把方案渲染出来，让你在浏览器里能看。

以前这个小页面完全没有认证——只要你的端口暴露在网络上，任何人拿到 URL 都能看到你正在讨论的方案，甚至还能往里注入事件，Agent 会当成是你的指令来执行。

v6.0 给这个 Web 页面加了一套完整的安全模型：

  1. 每次会话一个一次性密钥，嵌在 URL 里，浏览器存成 tab 作用域里的 cookie，所有请求和 WebSocket 连接都得带上。
  2. 文件服务器加了沙箱：拒绝符号链接、拒绝 .. 路径穿越、拒绝 dotfiles。
  3. 只在「图示比文字更有用」的时候才弹出来，并且尊重你「拒绝」的选择。
  4. 支持断线重连：Agent 一边跑，你一边看；页面断了会自动重连。
  5. 空闲超时从 30 分钟延长到 4 小时。

打个比方：以前这个看板就像把开会讨论的白板摆在马路边，路人都能看、都能上去擦两笔；现在给会议室装了门禁卡——一次性发卡，进会议室要刷卡，外人进不来。

5. v6.0：一套 Skill，全平台通用

Superpowers 以前有一套「Claude 腔」——Skill 里写的是「用 Task 工具」「写进 CLAUDE.md」。v6 把这些措辞全部换成了动作描述（「调度一个子代理」「你的指令文件」），然后给每个平台单独维护一份「动作 → 工具」的映射表。

图 7：v6 之后，同一套 Skill 通过 per-harness 的工具映射跑在 9 个以上的编码代理上。换个 Agent 工具就像换个手机——通讯录还是那套通讯录，只是同步方式不同。

6. 五个「防作弊」细节，才是 v6 最硬核的地方

把上面 SDD 的变化摊开看，v6 其实加了五个互相独立又互相补位的「防作弊」措施。单独看每一个都像小改动，合在一起才是「同质量下省一半 token」的真正原因。

图 8：v6 加的五道防作弊措施。官方在真实运行里抓到过 controller 教 reviewer 跳过某个 finding、结果缺陷真的流进主干的案例——所以现在「suppressing findings」和「pre-rating severity」被明文禁止。

用一个生活化的比喻总结这五道关卡：你让实习生独立改一份作业，以前是「他改完自己说改好了就交」，现在是「他改完必须把改动的每一行单独贴出来（diff 走文件）、不允许他提前跟质检员套近乎（controller 禁言）、质检员只能看不能动他的作业（reviewer 只读）、开工前先把任务清单对一遍有没有自相矛盾（plan 预检）、而且每派一个活都要写清楚派给谁（声明模型）」。哪一道都不是花架子。

7. v6.1：更瘦的 bootstrap、更严谨的打包

v6.1.0 和 v6.1.1 主要是「修自己」的工作。

7.1 Bootstrap 瘦身

using-superpowers 是每个会话都会注入的启动说明，它的体积直接影响每个会话的 token 支出。v6.1.0 做了几件事：

  • 删掉了 graphviz 画的 skill-flow 图（用文字替代）
  • 合并了 Instruction-Priority 和 User Instructions
  • 删掉了各平台的「How to Access Skills」详细教程
  • 裁剪 per-harness tool-mapping 文件（把已经和现代 Agent 默认行为重复的说明删了）

结果是：功能不变，但每个会话「入场费」降了。

7.2 v6.1.1：Codex hook 又踩了个坑

v6.1.0 本来想把 Codex 的 SessionStart hook 去掉——Codex 自己原生触发 Skill 已经很好用了，加 bootstrap hook 反而让体验变差。于是把 hook 配置删了。

结果反而更糟：Codex 在 hooks 字段缺失的时候会自动走 fallback 发现逻辑，把 repo 根底下给 Claude Code 准备的 hook 又注册上去了，还附带装一次性的 trust prompt。

解法非常「玄学」：manifest 里声明一个精确的空 hooks 对象 hooks: {}。Codex 读到这个值才会真正明白「没有 hooks」；如果字段缺失、给 []、或者给一个空内联 list，Codex 都会走 fallback。

这种 bug 你没见过一次都想不到——API 规范里「空」和「无」居然不是同一件事。打个比方：这就像你跟快递员说「我门口没有包裹要寄」，他听不懂；你必须递给他一张写着「本次寄件清单：空」的纸，他才真的不揽收。「没说」和「明确说了没有」在程序里是两回事。

7.3 v6.1.1：打包脚本

v6.1.1 还加了一个 package-codex-plugin.sh，给维护者打包 Codex portal 安装包用——.zip 或 tar.gz，时间戳归一化、保留可执行权限、验证每个 skill 都有 OpenAI metadata、dirty worktree 拒绝打包。

图 9：官方 v6.1.1 发布说明内容截图。hooks: {} 看起来只是一个空对象，实际决定了 Codex 会不会误加载另一套 SessionStart hook。

8. 一次发六个版本，节奏感拉满

从 v6.0.0 到 v6.1.1，一共打了 6 个 tag，跨度大约 16 天。把这条时间线画出来，能很直观地看到「大改 → 修安装 → 修安装 → 修路径 → 再大改 → 修细节」的典型开源节奏。

图 10：Superpowers 6 的发布节奏。v6.0.0 一口气重写了 SDD、Plans、脑暴看板和跨平台措辞；紧接着三个 patch 修安装和受保护路径；v6.1.0 再次瘦身 bootstrap 并接入 Codex 市场；v6.1.1 收尾 Codex hook 的空对象坑。

9. 怎么装、怎么验

不同 Agent 的安装入口不一样，但目标都是同一件事：让 Agent 在会话开始的时候能读到 using-superpowers，在任务触发的时候能加载对应 SKILL.md。

9.1 人工安装（通用）

去 Superpowers GitHub 仓库 看 README，找对应平台的章节。通常几条命令就搞定：

  • Claude Code：优先使用官方市场的 /plugin install superpowers@claude-plugins-official；也可以注册社区市场后安装
  • Claude Code 社区市场：/plugin marketplace add obra/superpowers-marketplace，然后 /plugin install superpowers@superpowers-marketplace
  • Codex：/plugins → 搜 superpowers → 安装
  • OpenCode：让 Agent 读取官方 .opencode/INSTALL.md 并按其中的原生插件方式安装
  • Cursor：/add-plugin superpowers
  • Pi：pi install git:github.com/obra/superpowers

9.2 一键全自动脚本（人工自动执行）

下面给三个平台的脚本，只依赖系统自带工具（git、curl、bash/zsh/PowerShell），不依赖任何第三方服务，只从官方仓库拉取。脚本会 clone 仓库到本地、打印当前版本、并给出对应平台的安装命令。

Ubuntu 26.04 / Linux（bash）：

#!/usr/bin/env bash
set -euo pipefail
DEST="${SUPERPOWERS_HOME:-$HOME/.local/share/superpowers}"
echo "==> 克隆 Superpowers 官方仓库到 $DEST"
mkdir -p "$(dirname "$DEST")"
if [[ -d "$DEST/.git" ]]; then
  git -C "$DEST" pull --ff-only
else
  git clone --depth 1 https://github.com/obra/superpowers.git "$DEST"
fi
VER="$(git -C "$DEST" describe --tags --always 2>/dev/null || echo unknown)"
echo "==> 当前版本：$VER"
echo "==> 仓库就绪。接下来根据你的 Agent 选择安装方式："
echo "    Claude Code : /plugin marketplace add obra/superpowers-marketplace"
echo "                  /plugin install superpowers@superpowers-marketplace"
echo "    Codex       : /plugins → 搜 superpowers → install"
echo "    OpenCode    : 在 opencode.json 加 \"plugin\":[\"superpowers@git+https://github.com/obra/superpowers.git\"]"
echo "==> 验证：问你的 Agent「你有哪些 Superpowers skills？」"

macOS 26（zsh）：

#!/usr/bin/env zsh
set -euo pipefail
DEST="${SUPERPOWERS_HOME:-$HOME/Library/Application Support/superpowers}"
echo "==> 克隆 Superpowers 官方仓库到 $DEST"
mkdir -p "${DEST:h}"
if [[ -d "$DEST/.git" ]]; then
  git -C "$DEST" pull --ff-only
else
  git clone --depth 1 https://github.com/obra/superpowers.git "$DEST"
fi
VER="$(git -C "$DEST" describe --tags --always 2>/dev/null || echo unknown)"
echo "==> 当前版本：$VER"
echo "==> 仓库就绪。Claude Code / Codex / OpenCode 安装命令同 Ubuntu 版本。"
echo "==> 验证：问你的 Agent「你有哪些 Superpowers skills？」"

Windows 11（PowerShell，无需 WSL）：

$ErrorActionPreference = "Stop"
$Dest = if ($env:SUPERPOWERS_HOME) { $env:SUPERPOWERS_HOME } else { "$env:USERPROFILE\superpowers" }
Write-Host "==> 克隆 Superpowers 官方仓库到 $Dest"
if (Test-Path "$Dest\.git") {
  git -C $Dest pull --ff-only
} else {
  New-Item -ItemType Directory -Force -Path (Split-Path $Dest) | Out-Null
  git clone --depth 1 https://github.com/obra/superpowers.git $Dest
}
$Ver = (git -C $Dest describe --tags --always 2>$null)
if (-not $Ver) { $Ver = "unknown" }
Write-Host "==> 当前版本：$Ver"
Write-Host "==> 仓库就绪。Claude Code / Codex / OpenCode 安装命令同上。"
Write-Host "==> 验证：问你的 Agent「你有哪些 Superpowers skills？」"

9.3 Agent 自动配置

你也可以直接让 Agent 自己装。把下面这段话发给你的 Agent 即可：

请帮我把 Superpowers 安装到我当前使用的 Agent 里。
步骤：
1) 识别当前 Agent 类型（Claude Code / Codex / OpenCode / Cursor / 其它）；
2) 用对应平台的官方安装命令安装 superpowers；
3) 安装完成后，列出已加载的 Superpowers skills；
4) 跑一个最小验证任务：让它用 systematic-debugging 的思路处理一个我给的小 bug，
   确认它真的「先收集证据再改代码」而不是上来就改。

9.4 验证它真的生效了

安装成功 ≠ 真的生效。五个验证动作：

  1. 直接问 Agent「你有哪些 Superpowers skills」？它至少应该能列出 using-superpowers、brainstorming、systematic-debugging、verification-before-completion。
  2. 给它一个小需求，观察它是不是先澄清目标或提出方案——而不是上来就写代码。
  3. 给它一个明确 bug，观察它是不是先收集证据——而不是上来就改配置。
  4. 让它必须运行真实验证命令再宣布完成——不能只说「应该可以」。
  5. 在多 Agent 场景里，不同 Agent 应该能说出同一套边界和检查点。

10. 常见问题 Q&A

Q: Superpowers 会不会拖慢我的 Agent？ A: Bootstrap 注入是每会话一次性的，Skill 加载是按需的。v6.1.0 还把 bootstrap 压缩过。真正拉慢节奏的是 Skill 触发的检查点——但那是你想让它慢的地方（比如写代码前先 brainstorming）。

Q: 我应该从哪个 Skill 开始用？ A: 只推荐三个起步：systematic-debugging（遇 bug 先证据后假设）、verification-before-completion（完成前必须真实验证）、writing-plans（大任务拆小步骤）。

Q: 我的模型很弱，装了还有用吗？ A: 越弱越有用。Superpowers 的本质是把模型「容易飘的部分」外置成流程。模型越强越能靠自己补流程；模型越弱越需要外部轨道把它扶住。

Q: 装了 Superpowers 就意味着不会出错吗？ A: 不是。它只保证流程被遵守，不保证方案最优、不保证测试覆盖 100%、不保证业务取舍正确。该做的 code review、日志审计、权限限制仍然要自己做。

Q: SDD 现在只用一个 reviewer，会不会漏问题？ A: 你担心的官方也担心过。所以他们同时加了：diff 以文件传递不丢信息、reviewer read-only 不改代码、controller 不允许暗示 severity、plan 预检暴露冲突、最终还有 whole-branch review 兜底。从 eval 看，同质量下更快更省，而不是「更凑合」。

Q: 为什么「空 hooks」和「没有 hooks 字段」不是一回事？ A: 这是 v6.1.1 踩的真实坑。Codex 的 manifest 里，字段缺失会触发 auto-discovery fallback，只有显式写 hooks: {} 才被当作「明确不要 hooks」。很多 API 都有这种语义——「没说」和「明确说了没有」不是一回事。

11. 这一轮升级，Superpowers 真正变强的是什么

回头看 v6.0 到 v6.1.1，真正值得记住的变化其实只有几个：

  1. SDD 变得更难绕过——一个 reviewer、两个 verdict、diff 走文件、controller 不许预设立场。
  2. Plans 有了结构——Global Constraints 和 Interfaces 把以前需要「意会」的约束变成「明文」。
  3. 脑暴看板加了一次性密钥——从「完全裸奔」到「有真正安全模型」。
  4. 跨平台统一——同一套 Skill，按平台映射工具。
  5. Bootstrap 瘦身——入场费降了。
  6. Codex 能从市场装——门槛低了。

Superpowers 没有给你承诺奇迹。它只给你一套可执行的工程纪律：对强模型，让输出更稳；对弱模型，让错误更早暴露；对多 Agent，让协作有共同语言；对真实项目，让「已完成」回到证据上，而不是停留在一句漂亮总结里。

如果你已经在用 OpenCode、Claude Code、Codex 或其他编码 Agent，我建议最低成本试一下：把 Superpowers 装上去，找一个低风险仓库跑几次 systematic-debugging 和 verification-before-completion，观察 Agent 是不是真的在「先想清楚再动手」。

祝调试顺利！
