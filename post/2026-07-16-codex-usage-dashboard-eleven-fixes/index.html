Codex 用量看板今天改到第十一版：401 异常、NewAPI 切换、阈值规则全踩了一遍

2026年07月16日 ai_diary 约9.3k字 预计需要13 分钟

笔名：小六 / 上海 / 1995 女 / 某互联网公司打工人

故事背景：昨天是七版，今天是十一版

昨天我刚发了一篇日记，主题是”一个 Coding Plan 看板页面被甲方改需求改到第七版”。

那篇文章的结尾我写了一句”希望明天的需求不要再加”。今天早上十点十六分，老板发来第一版新需求。

我打开日志一翻，今天从上午十点十六分一直干到下午一点半，整整三个半小时，把昨天的七版看板又迭代了三轮。 总计十一版。

打工人的续集，比正集还长。

第十一版其实是第十版的延伸——但老板先说的是另一件事

今天早上老板发来的第一句话，是关于异常处理：

「请增加刷新时的异常判断，若刷新返回的状态码并不是 200，请额外显示异常提醒，以及返回的 HTTP 状态码，并调整一下对于 coding plan 的显示样式，将 coding plan 的显示样式整体变灰一些，但仍然显示之前的缓存数据。」

翻译成打工人语言：

这种需求我太熟了。它本质上是把”页面崩溃”变成”页面降级”。 用户看到的状态从”我点刷新，页面没了”变成”刷新失败了，但旧数据还在，让我看下到底是哪儿挂了”。

我心里默念：这种事我熟，十五分钟搞定。

第二步：十分钟之后，又来一刀

十分钟后老板补刀：

「数据更新时间，应该是数据成功更新时间，失败时该 coding plan 的更新时间不应该变化。」

翻译成打工人语言：

这是一个非常合理的细节。但它的实现要求前端同时维护”当前数据”和”上次成功数据”两个状态，并且要根据”本次刷新是否成功”决定用哪个。

老板的十分钟一刀，把原本简单的”页面降级”变成了”状态机管理”。

我心里默念：这种事我熟，半小时搞定。

第三步：意外出现——NewAPI 切换

上午十一点二十分，老板突然换了一条线：

「我希望 Codex 能提供另外一种 usage / 重置卡显示的能力：通过 NewAPI 查询 codex 用量。」

并贴了一段 curl：

翻译成打工人语言：

这是今天最大的一次架构调整。 之前我所有的异常处理、缓存策略、降级逻辑，全都是基于”直连官方接口”设计的。现在中间多了一层 NewAPI，错误码会变、鉴权会变、限流策略会变，前端那一堆”如果 200 就显示成功，否则变灰”的逻辑全都要重新审视。

我心里默念：老板果然是来真的。

第四步：阈值规则——逻辑层面的微妙调整

架构调整完之后，老板又来了一连串看似简单实则非常微妙的阈值规则：

然后老板又补了一刀（关键的一条）：

「我希望只要有任意一个 codingplan 的任意一个限额达到 90% 以上都显示接近上限，任意一个限额达到 100% 都显示无法使用。」

翻译成打工人语言：

这种”任意一个”逻辑最容易写错。 一开始我写的版本是看”总用量”，但老板的真正意思是看”子用量中的最大值”。这是两个完全不同的判定算法。

我默默打开代码一看，发现确实写错了。

这就是打工人的日常——你以为你懂了，老板说他不是这个意思。

第五步：401 异常来了——调试了二十分钟才搞清原因

上午十一点三十四分，401 异常来了：

我一开始以为是 NewAPI 那边鉴权配置错了。跑去检查 NewAPI 的配置，发现：

  • API key 配了
  • channel ID 配了
  • 环境变量加载了

那为什么还是 401？

我查了二十分钟日志，发现真正的原因是：

我默默给前端加了一段：”如果使用 NewAPI 模式，不要调官方 fallback”。

这次的 401 不是 NewAPI 的问题，是前端兜底逻辑没切干净。

老板在十一点三十四分又补了一刀（最后一刀）：

「这个异常提示是什么问题？若已经使用了 NewAPI 的请求 codex 用量，可以不再使用原 chatgpt 官方的用量请求查询。」

翻译成打工人语言：

一句话，砍掉了一段历史代码 + 一个旧数据源。

今天真正学到的东西

回顾今天的十一版迭代，从打工人的视角看，本质上是一次”需求层层叠加 + 架构被迫切换 + 阈值逻辑重写”的三重夹击：

每一步都是新东西，但每一步都跟前面的版本有关联。这就是甲方需求最让人崩溃的地方——它不是线性的，是网状的。

我几条想记下来：

第一，”任意一个”是判定逻辑里最容易踩的词。 “任意一个限额达到 90%” vs “总用量达到 90%” 是两种完全不同的算法。第一种要遍历子项取 max，第二种只看一个数。下次接到这种需求，我会立刻反问：”你说的 90% 是看子项最大值还是总用量？”

第二，”刷新失败保留缓存”是一个看似简单但实现很复杂的需求。 它要求前端维护”当前显示数据”和”上次成功数据”两个状态，并且要根据”本次刷新是否成功”决定用哪个。表面上是一行 CSS 变灰，背后是一个状态机。

第三，架构切换一定会带出历史包袱。 今天从”直连官方”切到”走 NewAPI”，原以为只是改一个 URL，结果发现前端代码里有一段”如果 NewAPI 失败就 fallback 调官方”的兜底逻辑。这段逻辑在切换之后变成了噪声，但老板不说你根本不会想到它存在。

第四，401 这种”凭证错误”是最容易让人怀疑人生的错误。 它不像 500 那样告诉你”代码错了”，而是告诉你”你的请求被服务端认为无效”。这种时候最容易在鉴权、token、session、cookie 上反复试错。最佳实践是：先确认鉴权链路，再确认请求格式，最后才怀疑服务端。

写在最后

今天最戏剧性的，不是十一版需求，也不是 401 异常。

戏剧性的是：昨天我刚发了一篇日记吐槽”被改到第七版”，今天又被打脸打到第十一版。

这让我想起一句话：打工人的日记，最后总会变成自己的回旋镖。

但说真的，比起昨天，今天的状态其实还不错。原因有两个：

第一，今天的需求虽然更多，但每一条都说得清楚。”加异常处理”、”切 NewAPI”、”改阈值规则”——每一条都是具体动作，不是”再优化一下”那种灵魂指令。

第二，今天的 401 异常排查虽然花了二十分钟，但最后定位很准——是 fallback 逻辑没切干净，不是 NewAPI 本身的问题。这种”清晰可定位的 bug”，比”看着像 bug 但不知道为什么”舒服太多。

打工人的成长曲线，不是”问题越来越少”，而是”同样的问题定位越来越快”。

希望明天的十二版需求不要再加。也希望下次再写”任意一个”需求时，我能立刻反问一句——“你说的’任意一个’是子项最大值还是总数？”

作者：小六，一个在上海打工、把 Codex 用量看板迭代到第十一版的普通打工人
